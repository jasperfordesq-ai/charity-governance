## Cross-Route Findings

### Theme Scope

- `docs/architecture/09-frontend.md` and code show `(marketing)` and `(auth)` layouts force `className="light"` and `colorScheme: 'light'` in `apps/web/src/app/(marketing)/layout.tsx` and `apps/web/src/app/(auth)/layout.tsx`.
- Root layout currently applies dark mode only to app routes per existing architecture notes, so Phase 4 should make theme pre-paint universal.
- Dashboard pages already contain many `dark:` classes, but each page hand-rolls surfaces. Move to semantic tokens/components before route polish.

### Navigation And Workflow

- Protected route prefixes are centralised in `apps/web/src/lib/protected-routes.ts`; keep this as the route authority.
- Dashboard navigation exists in `apps/web/src/app/(dashboard)/layout.tsx`, but workflow ordering should better reflect trustee tasks: organisation setup, compliance year, evidence/documents, board, registers, deadlines, regulator readiness, export.
- Marketing/auth surfaces should not feel like a separate product once universal theming is added.

### Manual SVG And Icon Use

- `rg "<svg"` found many inline icons across marketing, auth, dashboard, and shared components.
- Replace control icons with a shared icon strategy, preferably lucide icons where available, while keeping brand/social icons deliberate.
- Icon-only controls need accessible names/tooltips.

### Large Files

- Highest-risk route files by line count: `registers` 804, `documents` 624, `board` 525, `dashboard` 511, `export` 451, `compliance/[principleId]` 397, `organisation` 390.
- These should be migrated first into shared primitives and focused child components. Avoid broad refactors outside each route family.

### Decorative Gradients/Orbs

- `apps/web/src/app/(marketing)/page.tsx` contains a gradient hero and decorative blurred rounded orbs.
- `apps/web/src/app/(marketing)/features/page.tsx`, `pricing/page.tsx`, `blog/page.tsx`, and `blog/[slug]/page.tsx` use gradient sections.
- Revamp direction should remove decorative orbs and avoid one-note gradient-heavy composition; trust-led SaaS surfaces should be denser and calmer.

### Page And State Gaps

- Many app routes already implement loading, empty, and error states locally, but the patterns are inconsistent.
- Client-side validation is strong for auth due shared schemas; many operational dashboard forms rely on server validation plus disabled required fields.
- `APPROVED` signoff UI/server validation currently covers board meeting date, minute reference, and approver name, but not explanation readiness for `NOT_APPLICABLE`/`EXPLAIN`.
- The future design system should include `LoadingState`, `EmptyState`, `ErrorState`, `LockedFeatureState`, `ReviewWarningState`, `StatusChip`, `SourceBadge`, `ProfessionalReviewFlag`, `DataList`, `FieldGroup`, and `StickyFormActions`.
